01

One screen

An agent hits a fork. Instead of stopping dead or guessing wrong, it parks the question and keeps working on everything still unblocked — and you answer when you're ready.

V3 is honest about what that requires: one screen. A smart inbox. Everything else folds into it.

the surface

The inbox

An async queue of asks — questions, proposals, decisions the agent parked. Auto-ranked by what each is blocking. You answer; the agent moves.

the detail, inline

The blocking view

Not a separate screen. Each card wears its blast radius — "blocks 3 tasks" — and expands in place into an impact tree showing exactly what your silence is stalling.

Triage flag for you. You asked to "root L2 in inbox items." Jobs pushed further: "You don't have two surfaces — you have one surface and one detail view. Embed the blast radius in the card. You never leave the inbox, or you'll grow it back into a project plan." I took that — the blocking view is now an inline expansion, not a navigable screen. This is the one place I reinterpreted your instruction; if you wanted a standalone blocking screen, say so and I'll restore it as a second surface.

03

The substrate — four tables, simpler now

Tables are authoritative; the event log is append-only audit/history only. With phase gone, the node has a single stored lifecycle and two computed signals.

  project ──┐ (1:N)
              ▼
  node ◄──── self edge: parent_id  (soft hierarchy)
   ├ id, project_id, parent_id
   ├ kind  goal│plan│step│task     — display hint, levels skippable
   ├ title, body
   ├ status  DRAFT│ACTIVE│DONE│DISCARDED   — the ONLY stored lifecycle
   ├ version                              — optimistic-concurrency guard
   ├ ✗ phase  — REMOVED in v3
   └ blocked, completeness  →  COMPUTED, never written
              │ (1:N)
              ▼
  ask  (the unit of human↔agent exchange — first-class child)
   ├ type  QUESTION│PROPOSAL│DECISION
   ├ state OPEN│ASSUMED│ANSWERED│CONFIRMED│OVERTURNED│DISCARDED
   ├ required  bool        — does an OPEN ask block its node?
   ├ options[] / assumption / answer
   └ links[]               — GitHub PR/MR, commit, external ref

  event  (append-only · history — feeds get_context, NO human screen)
   └ project_id, seq, actor(human│agent), verb, ref, payload, ts
FIG. 1 — the blast radius the inbox ranks by is computed from ask.required + the parent_id / dependency edges below a node.
04

Status & concurrency

  STORED ─ the only lifecycle             ASK lifecycle (the deferral engine)
  ┌──────────────────────────┐           ┌────────────────────────────────────────┐
  │ DRAFT ─▶ ACTIVE ─▶ DONE   │           │ OPEN ──answer──▶ ANSWERED                │
  │   └──────▶ DISCARDED(why) │           │  └─assume─▶ ASSUMED ─confirm─▶ CONFIRMED │
  └──────────────────────────┘           │               └─overturn─▶ OVERTURNED    │
                                          │                    └▶ agent re-triages   │
  COMPUTED (never stored)                 └────────────────────────────────────────┘
   • blocked(node)      = ∃ OPEN required ask ∨ unmet dependency edge
   • completeness(node) = SKETCH│SHAPED│SPECCED   (derived from resolved fields/asks)
   • blast_radius(ask)  = # of downstream nodes that unblock if this is answered
FIG. 2 — one stored axis, three computed signals. Illegal combinations are unrepresentable.
  • Optimistic concurrency. Every mutating MCP tool takes expected_version; stale writes are rejected with current state. Carried from the backlog tool's transition().
  • Per-project monotonic event.seq gives a total history order and lets the UI request "everything since seq N" after a WebSocket reconnect.
  • Overturn race, resolved. Agent marks a node DONE on an ASSUMED ask; you overturn → version bumps → the agent's next write is rejected as stale → it re-reads, sees OVERTURNED, and re-triages instead of corrupting state.
  • Blast radius & blocked are materialized and invalidated by the events that can change them; the WebSocket pushes only the cards whose rank actually moved.
05

Multi-harness — MCP is the only hard dependency

The loop works the same on Claude Code, Codex, or OpenCode. The MCP instructions field at initialize — consumed by all three — tells any fresh session what to do, with no per-harness file.

  any harness boots ──▶ MCP initialize (Streamable HTTP) ──▶
  ┌───────────────────────────────────────────────────────────┐
  │ Waypoint → InitializeResult.instructions:                  │
  │  "Connected to Waypoint. Call get_context(project) FIRST.  │
  │   Park forks with park_ask instead of guessing."           │
  └───────────────────────────────────────────────────────────┘
        ↑ portable — Claude Code, Codex, OpenCode all read this

        interactive — stays in Max plan, never headless -p

06

Cloud architecture

One TypeScript service, two protocols: MCP (Streamable HTTP) for agents, REST + WebSocket for the browser. Postgres behind both. Auth deferred, but project_id on every row is the future tenant boundary.

   ┌─────────────┐   Streamable HTTP / MCP    ┌───────────────────────────┐
   │  agents     │ ─────────────────────────▶ │   Waypoint API service     │
   │ CC·Codex·OC │ ◀───── tool results ─────── │   (TypeScript · Node)      │     ┌────────────┐
   └─────────────┘                            │   • MCP endpoint           │ ──▶ │  Postgres  │
   ┌─────────────┐    WebSocket (rank deltas) │   • REST/RPC for web UI    │     │  4 tables  │
   │  web inbox  │ ◀═══════════════════════════│   • event emitter + cache  │     └────────────┘
   │ (browser)   │ ─────── REST mutations ───▶ │                           │
   └─────────────┘                            └───────────────────────────┘
                          [ later, additive ] auth / tenant gate — keyed on project_id
FIG. 4 — TypeScript end to end keeps MCP tool types, REST DTOs, and UI props in one type system. Postgres for concurrent writers; Streamable HTTP because HTTP+SSE is deprecated (MCP spec 2025-03-26).
07

Projects & the aggregated inbox

  • Agent side: a project is chosen by working-directory mapping (with an env override) and passed on every tool call.
  • Human side: the inbox defaults to all projects, ranked by blast radius across the board; a project filter narrows it.
  • Tenant-ready: project_id on every row means the eventual auth layer gates on project membership with no schema change.

10

History — plumbing, not a screen

The append-only event log still exists, but with Track cut it is never a human surface. Its only job: feed get_context so the next agent session — on any harness — reads back the decisions without re-asking.

  EVENT LOG (no UI · compacted by get_context for the next session)
  seq 102  14:02  agent parked DECISION "clock source" on Replay  (blast=3)
  seq 103  14:09  human asked latency delta (thread)
  seq 104  14:11  agent answered ~40ns
  seq 105  14:16  human chose A → DEC-014, supersedes DEC-009
  seq 106  14:16  agent Replay DRAFT→ACTIVE, resumed; 3 tasks unblocked
       ↑ get_context returns a summary of this, not the raw rows
FIG. 9 — the history is for the agent's continuity, not a dashboard for the human to watch.
11

Open questions

Down to a handful — the product shape is settled.

  1. The reinterpretation. OK that the blocking view is an in-card expansion (Jobs' call) rather than its own screen? Restore a standalone blocking screen only if you disagree. Rec: keep it in-card.
  2. Blast-radius definition. Count only directly blocked nodes, or the full transitive downstream? Transitive is more honest but a bigger number that can mislead. Rec: show direct count, reveal transitive on expand.
  3. Assumption autonomy. May agents proceed-on-assumption on any ask, or only low-consequence/reversible ones (the backlog tool's HIGH/LOW gate)? Rec: low-consequence only; HIGH always blocks.
  4. Ship order. Inbox-only first cut (no impact tree), then add the in-card blocking view? Or both together? Rec: inbox first, blocking view fast-follow.
  5. Bootstrap reliability. Half-day spike to confirm the MCP instructions field reaches the model on all three harnesses before committing the spec? Rec: yes.
  6. GitHub linkage. Just store the PR/MR link on an ask, or poll for merged/unmerged state? Rec: store now, poll later.
