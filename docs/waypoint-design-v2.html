What changed from v1

V1 was a local-first Electron app for Claude Code. Your v2 directives deliberately reverse three of its core recommendations and add three new requirements. None of these are accidental drift — each is a stated decision.

01

The one thing — and the three layers

The Jobs review forced the thesis down to a single sentence, and it's the right one:

An agent hits a fork in the road. Instead of stopping dead or guessing wrong, it parks the question and keeps working on everything still unblocked — and you answer when you're ready.

Everything else is in service of that moment. Your three functional layers are not three products — they are three views over one shared substrate, ranked by how close they sit to that core moment:

L1 · the front door

Inbox

For the human: an async queue of what needs you. For the agent: a place to park asks, proposals, and decisions, and to watch blockers clear.

= the Asks system
L2 · the evidence

Map

Visualize the plan and progress; defer decisions; zoom roadmap → task; see what's pending and what's blocking — without leaving the decision context.

= the node tree + computed blocked / completeness
L3 · the memory

Track

Follow tasks through stages (research → plan → dev → test) and keep the full history — context for both the human and the next agent session.

= the phase swimlane + append-only event log
Triage note on the Jobs review. Jobs argued to delete L2 and L3 and ship only the inbox. You explicitly asked for all three layers and for multiple visualization options, so deletion would contradict a direct instruction. The synthesis I took: rank, don't delete. The inbox is the default surface and the signature interaction; Map and Track are reachable from it but never the front door. And the "multiple shots" are presented as a decision aid with a recommendation each — which answers his real objection, that undecided optionality is a smell.

03

One substrate, three views

The whole product is four tables. The three layers are read-projections (mostly) over them. Tables are authoritative; the event log is an append-only audit trail and history feed — derived from mutations, never a second source of truth. This is the single most important correction from the critic: pick one place where truth lives.

  project  ───┐ (1:N)
                 │
                 ▼
  node  ◄──────── self edge: parent_id  (soft hierarchy)
   ├ id, project_id, parent_id
   ├ kind  goal│plan│step│task   — display hint, levels skippable
   ├ title, body
   ├ status  DRAFT│ACTIVE│DONE│DISCARDED   — stored spine
   ├ phase   RESEARCH│PLAN│DEV│TEST          — stored, owned by L3
   ├ version  — optimistic-concurrency guard
   └ completeness, blocked  →  COMPUTED, never written
                 │ (1:N)
                 ▼
  ask  (the unit of human↔agent exchange — first-class child)
   ├ id, node_id, project_id
   ├ type  QUESTION│PROPOSAL│DECISION
   ├ state  OPEN│ASSUMED│ANSWERED│CONFIRMED│OVERTURNED│DISCARDED
   ├ required  bool      — does an OPEN ask block the node?
   ├ options[] / assumption / answer
   └ links[]  — GitHub PR/MR, commit, external ref

  event  (append-only · the History — §11)
   └ project_id, seq, actor(human│agent), verb, node/ask ref, payload, ts
        ↑ derived from every mutation; replay = audit, not state source
FIG. 1 — the four tables. L1 reads ask; L2 reads node + computed fields; L3 reads node.phase + event.

Why tables-authoritative, not event-sourced

  • Reads are trivial — no fold/replay to answer "what's blocked right now."
  • Matches the prior art (the backlog tool's ticket_transitions is an audit log beside authoritative rows).
  • History and current state can drift if a mutation forgets to emit an event → enforce emission in one write path.

The leak the critic caught

L3 Track is not a pure read-view: the swimlane writes node.phase. So v2 admits it — phase is a stored, writable axis that L3 owns. L1 and L2 stay pure projections. Honesty here prevents a "view that secretly mutates" bug class.

04

Status model — two stored axes, two computed

V1 floated three orthogonal writable dimensions. The critic was right that that is a combinatorial mess no one keeps consistent (DONE + SKETCH should be impossible). V2 collapses to two stored axes and two derived signals.

  STORED ─ spine (lifecycle)            STORED ─ phase (L3 swimlane, tasks only)
  ┌──────────────────────────┐         ┌────────────────────────────────────┐
  │ DRAFT ─▶ ACTIVE ─▶ DONE   │         │ RESEARCH ─▶ PLAN ─▶ DEV ─▶ TEST      │
  │   └──────▶ DISCARDED(why) │         └────────────────────────────────────┘
  └──────────────────────────┘

  ASK lifecycle (the deferral engine)
  ┌─────────────────────────────────────────────────────────────────────┐
  │  OPEN ──answer──▶ ANSWERED                                            │
  │   │                                                                   │
  │   └─proceed-on-assumption─▶ ASSUMED ──human confirms──▶ CONFIRMED     │
  │                                │                                      │
  │                                └──human overturns──▶ OVERTURNED       │
  │                                        └─▶ agent re-triages subtree   │
  └─────────────────────────────────────────────────────────────────────┘

  COMPUTED ─ never stored, derived on write, cached, invalidated by events
   • blocked(node)      = ∃ OPEN required ask  ∨  unmet dependency edge
   • completeness(node) = SKETCH │ SHAPED │ SPECCED
                          ↑ derived from which fields/asks are resolved
FIG. 2 — spine + phase are written; blocked + completeness are computed. Illegal combinations become unrepresentable.

Concurrency — the gap v1 never closed

Multiple agents and a human write the same graph asynchronously. V2 adopts the prior art's pattern and extends it:

  • Optimistic concurrency. Every mutating MCP tool takes expected_version; a stale write is rejected with the current state so the caller re-reads. Carried directly from the backlog tool's transition(expected_version).
  • Per-project monotonic event.seq gives a total order for history replay and lets the UI request "everything since seq N" after a reconnect.
  • The overturn race, resolved. Agent transitions a node to DONE on an ASSUMED ask; human overturns. The overturn bumps the node version, so the agent's next write is rejected as stale → it re-reads, sees OVERTURNED, and re-triages rather than corrupting state. This is exactly the scenario the critic flagged as silent.

Blocked, materialized

"Computed" is the semantic source of truth, but recomputing a graph cascade on every read — then pushing it over WebSocket — over-broadcasts. So the derived blocked value is cached on the node and invalidated by the events that can change it (ask opened/answered, dependency edge added/met). The UI receives only the delta for the nodes whose blocked actually flipped.

Multi-harness — MCP is the only hard dependency

The core loop must work the same whether the agent is Claude Code, Codex, or OpenCode. The research pass found the clean seam: the MCP instructions field returned at initialize is consumed by all three harnesses, so the server can tell any fresh session what to do with zero per-harness files.

Cloud architecture

Cloud-hosted, browser UI, one backend service. Auth is deferred but the project boundary is the future tenant boundary — every row already carries project_id, so dropping an auth layer in front later is additive, not a migration.

   ┌─────────────┐   Streamable HTTP / MCP    ┌───────────────────────────┐
   │  agents     │ ─────────────────────────▶ │                           │
   │ CC·Codex·OC │ ◀───── tool results ─────── │   Waypoint API service    │
   └─────────────┘                            │  (TypeScript · Node)       │
                                              │   • MCP endpoint           │
   ┌─────────────┐    WebSocket (live)        │   • REST/RPC for web UI    │     ┌────────────┐
   │  web UI     │ ◀═════ blocked/ask deltas ═ │   • event emitter + cache  │ ──▶ │  Postgres  │
   │ (browser)   │ ─────── REST mutations ───▶ │                           │     │  4 tables  │
   └─────────────┘                            └───────────────────────────┘     └────────────┘
                                                      │
                              [ later, additive ]  auth / tenant gate  ── keyed on project_id
FIG. 4 — single service, two protocols. MCP for agents, REST + WebSocket for the browser; one Postgres behind both.

Stack — recommended

BackendTypeScript / Node — one language with the web UI; mature MCP SDK
DBPostgres — concurrent writers, JSONB for ask options/payloads
TransportStreamable HTTP MCP + WebSocket to UI
UIReact (carries from backlog tool); axiom style

Why not Python + the old SSE transport

  • Two languages (Py backend + TS UI) doubles the surface for a one-person tool.
  • HTTP+SSE transport is deprecated as of MCP spec 2025-03-26 — building on it is starting in debt.
  • TS keeps MCP tool types, REST DTOs, and UI props in one type system.
07

Projects & the cross-project inbox

A project is the top of every tree and the unit an agent registers against. Agents pass project to get_context / park_ask; the human gets a switcher plus an aggregated inbox that answers the v1 requirement "one notification of everything pending across projects."

  • Agent side: a project is selected by a config value (e.g. WAYPOINT_PROJECT) or inferred from the working directory's registered mapping, then passed on every tool call.
  • Human side: the inbox defaults to all projects, sorted by what's been waiting longest; a project filter narrows it. The map and swimlane are always scoped to one project.
  • Tenant-ready: because project_id is on every row, the eventual auth layer gates on project membership with no schema change.

Open questions for v2 review

Fewer than v1 — most of v1's questions are now settled. These are the ones v2's reviews surfaced or left genuinely open.

  1. Ship order. The critic argued for shipping L1 Inbox alone first, then Map and Track once the model is proven. Do you want a phased build (Inbox → Map → Track), or all three layers in the first cut? Rec: phased — Inbox first.
  2. Soft hierarchy. Keep kind as a goal/plan/step/task display hint (recommended, keeps your vocabulary), or go fully free-form with just a level_hint as the critic suggested? Rec: keep kind as a hint.
  3. Phase axis scope. Does the research→plan→dev→test phase apply only to task nodes, or to plans/steps too? Rec: tasks only — higher levels use spine + completeness.
  4. Assumption autonomy. Should agents be allowed to proceed-on-assumption on any ask, or only on ones marked low-consequence/reversible (mirroring the backlog tool's HIGH/LOW gate)? Rec: only low-consequence; HIGH always blocks.
  5. Roadmap default. Confirm outline-tree as the primary map with timeline overlay (vs. making timeline the default). Rec: outline-tree primary.
  6. Bootstrap reliability. Do we spike the MCP instructions-field behavior across all three harnesses before committing the spec (research flagged it unverified)? Rec: yes — half-day spike.
  7. Project selection. Agent picks project via env var, working-directory mapping, or an explicit select_project tool call? Rec: dir-mapping with env override.
  8. History compaction owner. Should get_context summarize server-side (deterministic, cheap) or hand raw events to the agent to summarize (flexible, costs tokens)? Rec: server-side, with a raw-events escape hatch.
  9. GitHub linkage depth. v1 wanted PR/MR links on tasks. For v2: just store the link, or poll GitHub for merged/unmerged state? Rec: store link now, poll later.
  10. Still "Waypoint"? Name still fits the cloud, multi-harness framing. Rec: keep it.

Waypoint — product design v2 · 2026-06-13 · supersedes v1. Reply with the questions you'd answer differently; unanswered ones default to the recommendation. After this review locks, the next step is the implementation spec (interfaces & schema first, per the commit rules).
