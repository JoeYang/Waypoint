01

Thesis & the core loop

Every agent workflow today has the same failure mode: the agent hits a judgment call, asks, and stalls. The human context-switches in to unblock a single question, or worse, the session dies and the question dies with it. Waypoint inverts this: the human is modeled as a slow, asynchronous, high-value dependency — like a flaky external API you design around, not against.

The core loop:

  1. Agent uploads structure. During brainstorming/planning, the agent lodges the goal, sketches the roadmap, and progressively decomposes it. Nothing needs to be fully mapped — decompose only when you must.
  2. Agent lodges asks. When it hits a judgment call, it files a proposal (approve/revise) or a decision (pick an option), with a recommendation and consequences — then either works on something else that's unblocked, or proceeds on a stated assumption.
  3. Human answers in batches. Open the app, see everything waiting on you ranked by what it unblocks, answer, close the laptop.
  4. Agent resumes with context. The answer plus the full decision history is delivered back into an agent session, which continues — or re-triages if an assumption was overturned.

Secondary product: the byproduct of this loop is a permanent, structured history of how the roadmap evolved and why every key decision was made — which becomes the context pack for every future agent session. The backlog tool in ~/projects/tools/backlog proved the approval-gate half of this; Waypoint's new ground is the deferral half.

03

Hierarchy — three shapes

Your vocabulary: goal → plan → step → task → (proposal | decision). The question is whether that's a schema or a vocabulary.

H1 — Rigid typed levels

Five entity types, five tables, foreign keys enforce the ladder. A task can only live under a step.

Pros
  • Matches your stated mental model exactly
  • Per-level UI is trivial to specialize
Cons
  • Agents waste turns debating "is this a step or a task?"
  • Small efforts must invent fake plans/steps to hold one task
  • Schema migration every time the ladder changes

H2 — Fully emergent nesting

One item type, free nesting, no kinds. Hierarchy is whatever the agent builds (the Jobs position).

Pros
  • Zero taxonomy arguments
  • Simplest schema, simplest MCP surface
Cons
  • No shared altitude vocabulary between you and the agent
  • Roadmap view has nothing reliable to render at "goal level"

H3 — One node type, kinded recommended

A single node table with a kind field (goal/plan/step/task). Kind ordering is enforced softly (a task's ancestors must be ≥ step rank) but levels are skippable — a goal may directly hold tasks. Asks (proposals/decisions) attach to any node, as children of a different type (§04).

Pros
  • Your vocabulary, the agent's flexibility
  • One table, one MCP tool set, one tree component
  • Altitude views filter by kind — robust roadmaps
Cons
  • Soft rules need validation logic + good agent prompts

The completeness dimension

Per your requirement, every node carries a colored spec-completeness tag: sketch shaped specced — sketch = title and intent only; shaped = boundaries, approach, and risks named; specced = an agent could start implementing without asking anything new. Two sub-options (open question Q3):

  • Declared — the agent sets it when it raises the bar. Simple; risks drifting from reality.
  • Computed — derived: a node is specced iff it has no open spec-type asks and its required fields are filled; shaped iff decomposed with asks identified. Never lies, but the inference rules need tuning.

Discard-with-reason applies to any node: the reason is stored on the node (Jira's status≠resolution lesson), and all descendants plus assumption-tainted dependents flip to a STALE flag for agent re-triage — the agent proposes keep / rework / discard per affected node rather than the tool auto-deleting work.

04

State model — three proposals

Research swept Linear, Jira, GitHub Projects, Shape Up, Gerrit/kernel review, and human-in-the-loop patterns in LangGraph and Temporal. The load-bearing findings: Linear models blocked as a relation, not a status; Jira's durable lesson is status ≠ resolution; Gerrit scales with a tiny lifecycle spine + orthogonal labels; Temporal/LangGraph model waiting-on-human as a durable, queryable sub-thread that never freezes sibling work. Three proposals:

P1 — Monorail: one machine, waiting-as-state

DRAFT → READY → IN_PROGRESS ⇄ NEEDS_HUMAN → REVIEW → DONE (any) ──────────────────────────────────────────→ DISCARDED (reason)
Pros
  • Maximally legible; the human inbox is literally one column
  • Closest to the existing backlog tool
Cons
  • An item with one open question but three workable subtasks must lie about its state — the exact failure Waypoint exists to fix
  • Per-type differences become guard-clause warts

P2 — Spine + Asks recommended

One tiny lifecycle shared by all node kinds, plus orthogonal dimensions. Deferral is structural, not disciplined.

lifecycle spine (every node): DRAFT → ACTIVE → DONE └───────┴────→ DISCARDED (reason; descendants → STALE) completeness (every node): SKETCH → SHAPED → SPECCED phase (task nodes; agent-set): RESEARCH → PLAN → DEV → TEST Ask — first-class child object {kind: PROPOSAL | DECISION}: OPEN ──answer──→ ANSWERED (approved / option chosen / revise+suggestions) OPEN ──agent proceeds──→ ASSUMED ──human──→ CONFIRMED | OVERTURNED └→ tainted nodes → STALE blocked — COMPUTED, never stored: blocked(n) ⇔ ∃ required Ask OPEN on n ∨ ∃ unmet dependency edge into n
Pros
  • "Defer and keep working" is the data model: an OPEN ask blocks only what genuinely requires it
  • Human inbox = all OPEN/ASSUMED asks ordered by unblock count — one query
  • Steals each system's validated idea: Gerrit's spine, Linear's blocked-relation, Jira's resolution split, Temporal's durable waits
  • Assumption flow (§05) falls out naturally
Cons
  • Less glanceable than one status field — UI must render computed blocked-ness well
  • More concepts for agents to drive (mitigated by a tight MCP tool design)

P3 — Schema per type

proposal: DRAFT → PROPOSED → APPROVED | REVISE decision: OPEN → OPTIONS_READY → CHOSEN task: RESEARCH → PLAN → DEV → TEST → DONE shared terminal contract: DONE | DISCARDED(reason)
Pros
  • Each type maximally precise; Jira-proven approach
Cons
  • N machines to evolve; "what's waiting on me" reimplemented N times
  • Cross-type rollups need per-type adapters; config tax grows forever

Recommendation: P2. It is the only design where the product thesis is enforced by the schema rather than by agent discipline. P1 remains the fallback if P2's UI proves too subtle — they share enough vocabulary that downgrading later is a migration, not a rewrite.

The ask contract (carried from the backlog tool)

Every ask keeps the human-review contract that already works in backlog-tickets.md: Type & Ask → Situation (≤5 lines) → Options (each with consequences) → Recommendation → Consequences (if approved / if rejected or deferred / reversibility). One decision per ask. Decisions carry structured options [{key, label, summary}] rendered as buttons. New fields: required (does this block its node?), assumption (what the agent will do if you stay silent), and unblocks (computed blast radius).

Design rules: confirming is ⏎ on the focused inbox card. The assumption is always stated in the ask before the agent proceeds (no retroactive assumptions). Required asks with no safe assumption stay OPEN and genuinely block — "defer unless we must decide" cuts both ways.

06

Answer delivery — the hard problem

Your question: "is it possible to inform agents via SSE using MCP?" Platform research gives a clear answer: not today. The MCP spec supports server→client notifications over SSE/streamable HTTP, but Claude Code currently ignores them — they don't re-invoke the model (tracked and closed as not-planned in claude-code#36665). And the deeper truth (the Jobs point): sessions end. The durable primitive can't be "a live session notices" — it must be "an answer produces a resumable context pack, and some trigger gets an agent to consume it." Four trigger mechanisms, composable:

MechanismHow it worksMaturityLimitation
Poll between tasksAgent calls waypoint.sync() (one MCP tool returning all answers + new STALE flags since last sync) at natural boundaries — task completion, session start.Works today, zero infraLatency = task length; only while a session lives
ChannelsA custom Waypoint channel pushes "ask answered" events into a running session (Claude Code ≥2.1.80, research preview, --channels).Research previewSession must be alive; preview-stage API
Session event pushWaypoint backend tracks session IDs per ask; on answer, POSTs a user.message event via the Managed Agents Event API — re-invokes the model mid-session.Production (managed agents)Requires session-ID bookkeeping; managed-agent deployments
Fresh-session spawn the moatOn answer (or batch of answers), Waypoint invokes claude -p headless with the context pack: the answer, its rationale, the node's decision history, and a resume cursor. Works when nothing is running.Works todayCold start cost; context pack quality decides everything

Recommended architecture: design everything around the context pack + sync() (mechanisms 1+4 — durable, works today), and treat live push (2/3) as a latency optimization layered on later. This makes the context pack — "everything a fresh agent needs to resume this node" — a first-class API object, which §08 turns into the history product. Open question Q4: whether v1 includes auto-spawn or just makes resume-by-hand trivially easy.

UI architecture — three options + sketches

Research swept Linear (Initiatives→Projects→Issues + the industry's best Inbox), GitHub Projects/Height (views as projections), Jira Advanced Roadmaps (dense nested Gantt), and agent-supervision UIs (Devin's command center, Vibe Kanban). Notable: nobody yet combines altitude navigation with an approvals queue — that's Waypoint's opening.

Option A — Zoomable single canvas

[zoom: ▣ goal  ▢ plan  ▢ task]                    ⚠ 3 on YOU
 ┌─Goal: Auth ───────────┐        ┌─Goal: Billing──┐
 │ ┌plan A┐ ──▶ ┌plan B┐ │ ─────▶ │  ▣▣▢   agents×2 │
 │ └──────┘     └─⏸YOU─┘ │        └────────────────┘

One infinite canvas; semantic zoom changes altitude; dependencies are edges. Pros: dependencies first-class, spatial memory, demos beautifully. Cons: heaviest build (canvas perf, auto-layout); poor for dense status scanning; spatial tools historically lose to lists for tracking. Verdict: a beautiful v3, wrong v1.

Option B — Altitude tabs with drill-down (the Linear model)

Separate pages per altitude, joined by drill-down; a global Inbox tab. Pros: proven, cheapest, boards fit agent fleets naturally. Cons: altitude context evaporates on drill-down; the Inbox is disconnected from the hierarchy — you see that you block something, not what subtree a decision unblocks.

Option C — Inbox-first + structure spine recommended

The research recommended a tree-first spine with an attention dock; the Jobs review argued the inbox is the product. They compose: the home screen is the decision inbox (cross-project, ranked by blast radius); selecting an item reveals the tree as the evidence pane, with the blocking path highlighted — you always see exactly what your answer unblocks. The full tree/board/timeline views are one click away as the "structure" surface.

08

History as agent context

Everything is append-only events: node created, decomposed, ask lodged, assumption stated, answered, overturned, discarded-with-reason, PR linked, phase moved. Three products fall out of one log:

  • Timeline per node — the human-readable audit trail (the backlog tool already proved this pattern).
  • The context pack — get_context(node_id) returns a compact, agent-optimized brief: the node's position in the hierarchy, every ancestor decision with chosen option + rationale, active assumptions it's riding on, and open asks in scope. This is what makes a fresh session as informed as the one that lodged the work — the answer-delivery moat (§06) and the "history for agents later" requirement are the same feature.
  • Decision archaeology — "why did we choose Kafka?" is a query, not a memory. Superseding a decision links old → new, never edits history.
09

GitHub, multi-project, notifications

  • PR linking: tasks carry PR references (agent links them via MCP at DEV phase). Waypoint shows live merge status — a periodic gh/API poll in v1, a webhook later. The structure view thus doubles as "what's merged vs. in flight," your requirement #5. A merged PR nudges (not forces) the task toward TEST/DONE.
  • Projects: agents register a project over MCP (register_project(name, repo, path)); the human switches projects in the app, and the inbox aggregates across all of them by default — the "overall notification" you asked for. Per-project views are one filter away.
  • Notifications: OS-level (Electron) notification when a new required ask arrives or an assumption is about to be acted on; configurable quiet hours. The inbox badge is the source of truth; notifications just point at it.

Backend

Shape either way: a single local service process owns the DB and exposes (a) HTTP + WebSocket/SSE for the Electron UI, (b) streamable-HTTP MCP for agents — so multiple Claude Code sessions across projects hit one server, which is what makes cross-project aggregation work. The Electron app supervises the service (starts it, shows its health). Worth noting: this also leaves the door open to a plain browser tab and a phone later — the Electron shell is packaging, not architecture.
